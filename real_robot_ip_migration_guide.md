# 项目后期改为 IP 通信控制真机机械臂指导文档

## 1. 文档目的

本文档用于指导后期把当前 `GNN_solver` 项目，从 MuJoCo 仿真执行，逐步迁移为通过 IP 通信控制真机机械臂。

本轮只整理方案，不修改现有功能代码。

## 2. 当前约束与原则

### 2.1 当前明确约束

1. 通信方式采用 IP 通信。
2. 发送内容以字符串形式传递给机械臂。
3. 传递内容以每次求解迭代结束时的关节角度为主。
4. 机械臂侧还需要同时使用 `a` 和 `v` 参数。
5. 夹爪子任务、目标夹取细节暂时不处理。
6. 后续改造遵循“小改优先，如无必要勿增实体”。

### 2.2 本文默认假设

1. 当前主机械臂对象是 UR10e，或至少接口风格与 UR 系机械臂接近。
2. 真机阶段先解决“机械臂本体到位”，暂不接入夹爪控制。
3. 当前仿真中的逆解、子任务拆分、行为树流程尽量保留。

如果后期真机不是 UR 系，而是其它品牌，本文中的“发送字符串格式”和“机器人端执行方式”需要替换，但整体改造分层仍可复用。

## 3. 当前工程中最适合的最小改动接入点

结合当前工程，后续真机改造最相关的代码路径如下：

1. `GNN/ik_solver_api.py`
   这里的 `GNNIKSolver.solve(...)` 是逆解核心迭代入口。
   如果你坚持“每次求解迭代结束都发送一次关节角”，最终最直接的接入点就在这里。

2. `subtask/task_linear.py`
   当前直线插补逻辑会循环调用 `solver.solve(...)`，每一步都得到新的关节角。
   如果后期发现“每个 solver 内部迭代都发一次”过于频繁，那么这里是最自然的降频发送点。

3. `subtask/task_reach.py`、`subtask/task_approach.py`、`subtask/task_orient.py`、`subtask/task_retreat.py`
   这些任务本质上还是通过 `task_linear` 或 `solver.solve` 间接得到关节角。
   它们本身不应承担通信职责，尽量保持不动。

4. `subtask/task1_bt_demo.py` 等行为树 demo
   这些脚本更适合作为“切换仿真模式/真机模式”的入口，不建议在这里堆通信细节。

## 4. 推荐的总体改造思路

推荐把后续真机改造分成两层：

1. 上层仍然保留你当前的“视觉/子任务/行为树/逆解求解”流程。
2. 下层只新增一个很薄的“机械臂发送适配层”，负责把关节角转成字符串，通过 IP 发给真机。

这样做的原因是：

1. 不动你已经验证过的 GNN 求解逻辑。
2. 不把网络发送逻辑散落到各个子任务中。
3. 后期如果仿真/真机要切换，只切换“执行出口”即可。

## 5. 强烈建议的发送方案

### 5.1 推荐方案：直接发送可执行的 URScript 字符串

如果真机确实是 UR10e，最省改动的方案不是自定义复杂协议，而是直接从 PC 端发一条字符串：

```text
movej([q1,q2,q3,q4,q5,q6], a=0.15, v=0.08)
```

再加上换行：

```text
movej([2.100000,-1.520000,1.480000,0.020000,1.560000,-0.010000], a=0.15, v=0.08)\n
```

这套方式的优点：

1. 本身就是字符串，符合你的要求。
2. `q`、`a`、`v` 都能直接表达。
3. 机器人端不需要再额外解析 JSON 或自定义字段。
4. 对现有工程来说新增实体最少。

### 5.2 不太建议的方案：自己定义 JSON/CSV 协议让机器人端解析

例如：

```json
{"q":[...],"a":0.15,"v":0.08}
```

这个方案在上位机很好处理，但很多机械臂控制器原生字符串解析能力一般，反而会让机器人端更复杂。  
在“尽量小改”的原则下，不作为第一推荐。

## 6. 关于“每次求解迭代结束都发送一次”的现实提醒

这个想法从算法视角是成立的，因为你确实希望把求解过程中的关节角逐步输出给真机。  
但从真机执行视角看，它不是“能不能发”的问题，而是“发出去后机器人是否能稳定、可控、可复现地执行”的问题。

### 6.1 为什么仿真能跑，真机却可能不稳

在仿真里，`GNNIKSolver.solve(...)` 的迭代和 MuJoCo 状态更新都发生在同一进程里：

1. 你更新一次 `angles`。
2. 你立刻把这组角度写入仿真。
3. 你立刻看到新的状态。

但在真机里，中间多了至少三层现实因素：

1. 网络传输有延迟和抖动。
2. 机器人控制器有自己的接收和执行节拍。
3. 一条运动指令通常不是“瞬时生效”，而是需要一段时间执行。

所以，求解器里每迭代一次就发一次，并不等价于真机会“按你的每次迭代平滑跟随”。

### 6.2 当前工程下，发送频率可能远高于你直觉

这个问题在你当前工程里尤其明显，因为发送频率不是单层循环，而是“外层插补 * 内层求解迭代”的乘积。

以现有默认参数估算：

1. `subtask/task_linear.py` 默认 `INTERPOLATION_STEPS = 20`
2. `GNN/ik_solver_api.py` 默认 `MAX_ITER = 50`

这意味着一次线性动作，理论上最多可能触发：

```text
20 * 50 = 1000 次角度更新机会
```

如果这些更新都直接变成真机命令，那么一个本来只是“从 A 点到 B 点”的动作，就可能在极短时间内塞给机器人数百到上千条指令。

### 6.3 真机上最容易出现的几类问题

#### 6.3.1 指令堆积

如果你发送速度大于机器人执行速度，控制器侧就可能出现命令排队。  
一旦排队，后面的指令虽然“已经发出”，但执行时机已经落后于上位机当前判断。

结果就是：

1. 上位机认为机器人已经接近目标。
2. 机器人实际上还在补执行旧命令。
3. 两边状态开始脱节。

#### 6.3.2 机械臂抖动或来回修正

求解器内部迭代本身是为了收敛，它输出的中间角度未必都适合直接执行。  
有些中间结果只是“朝正确方向迈一步”，并不是物理上理想的轨迹点。

如果把这些中间量全部下发，真机可能表现为：

1. 小幅来回摆动。
2. 某几个关节高频微调。
3. 接近目标时抖动更明显。

#### 6.3.3 上一条未完成，下一条已经覆盖

对于真机来说，一条新的位置命令很可能会改变当前运动计划。  
如果在极短时间内连续发送，机器人表现可能不是“严格依次执行”，而是不断被新目标重定向。

这会导致：

1. 实际轨迹被压缩或扭曲。
2. 你以为在做平滑逼近，实际上机器人在不断抢最新目标。
3. 最终到位效果不一定比低频发送更好。

#### 6.3.4 仿真与真机不一致

MuJoCo 里你是“写入关节角后直接得到结果”，但真机有加速度、速度、控制器插补、机械惯性、实际负载等因素。  
因此，同样一串角度，真机实际走出来的轨迹通常不会和仿真逐点重合。

### 6.4 什么时候“每次迭代都发送”才比较合理

不是绝对不能做，而是要满足至少下面几个条件：

1. 机器人端接收机制足够稳定。
2. 发送频率被严格限制。
3. 只有有效变化才发送。
4. 发送失败或网络断开时能安全停机。
5. 上位机和机器人端最好有某种简单确认机制。

换句话说，这更像一个“受控流式更新”问题，而不是简单的“有新角度就立刻发”。

### 6.5 推荐的两阶段推进方式

#### 阶段 A：先做外层插补发送

优先在 `subtask/task_linear.py` 这一层发送，也就是每完成一次插补点求解，再发一次当前关节角。

这个阶段的核心目标不是追求最细粒度，而是先确认：

1. IP 通信链路稳定。
2. 真机能正确接收字符串。
3. 低速参数下动作方向正确。
4. 关节顺序、单位、限位都没有问题。

这个阶段最适合做：

1. 首次联调。
2. 限位验证。
3. 坐标理解验证。
4. 异常断连处理验证。

#### 阶段 B：再下沉到 solver 内部迭代发送

等阶段 A 稳定之后，再考虑把发送点下沉到 `GNN/ik_solver_api.py` 的迭代循环内部。  
此时的目的不是“让它发得更多”，而是“让它更连续，但仍然受控”。

### 6.6 如果下沉到 solver 内部，至少要加哪些控制

如果后面你坚持按原设想做“每次迭代都发”，建议至少加下面 5 类控制。

#### 6.6.1 最小发送周期

不要每轮循环都无条件发送，应该设定一个最小间隔，例如：

1. 距离上次发送不足 `min_send_interval` 时，不发。
2. 等到时间满足后，再发送最近一次有效角度。

这可以避免求解器迭代速度远高于真机接收速度。

#### 6.6.2 角度变化阈值

如果本轮角度和上次发送角度相比变化极小，就没必要发。  
建议至少检查：

1. 单关节最大变化量。
2. 六关节整体范数变化量。

只有变化超过阈值才发送。

#### 6.6.3 收敛末期降频

越接近目标，越容易出现高频微调。  
因此在误差进入小范围后，建议自动降低发送频率，而不是继续高频更新。

简单做法：

1. 远离目标时允许较快发送。
2. 接近目标时只保留关键更新点。

#### 6.6.4 异常后停止继续发送

一旦出现：

1. socket 断开
2. 连续发送失败
3. 数据格式异常
4. 超出限位

就应该立即停止继续发送，并把控制权交还给上层，而不是悄悄忽略错误继续求解。

#### 6.6.5 最好保留“只求解不发送”模式

即使将来支持真机流式发送，也必须保留当前纯仿真模式。  
这样当真机联调异常时，你可以快速切回仿真验证问题到底出在：

1. 逆解本身
2. 通信层
3. 真机执行层

### 6.7 推荐的控频规则

后续实现时，我建议优先采用下面这种简单规则，而不是一开始就做复杂协议：

```text
仅当满足“时间阈值 + 角度变化阈值 + 安全检查通过”时才发送
```

也就是：

1. 时间上不过密。
2. 角度上不是无意义的小抖动。
3. 发送前再次确认仍在安全范围内。

### 6.8 一条非常务实的判断标准

如果后面真机出现下面任意一种现象，就说明“每次迭代都发”当前不适合继续推进：

1. 机械臂明显发抖。
2. 明明速度参数很低，但动作依然不顺。
3. 停止上位机后，真机还在继续执行一串旧动作。
4. MuJoCo 看起来正常，真机轨迹却明显异常。

这时不要硬扛，直接回退到“外层插补发送”就是最稳妥的做法。

## 7. 后续最小改动实现建议

后面真正开始改代码时，建议把改动收敛到少数几个点，尽量做到“新增薄层，不重写主链路”。

### 7.1 目标不是重构工程，而是补一个执行出口

你当前项目的主链路已经比较清楚：

```text
视觉/任务目标 -> 子任务 -> solver.solve(...) -> 得到关节角 -> 仿真刷新
```

后续真机接入时，最小改法应该是把它扩成：

```text
视觉/任务目标 -> 子任务 -> solver.solve(...) -> 得到关节角 -> 仿真刷新
                                                        -> 可选 IP 发送到真机
```

也就是说：

1. 保留原仿真出口。
2. 旁路增加一个真机发送出口。
3. 默认仍然是“不发真机”。

### 7.2 建议新增一个独立的机械臂发送模块

建议只新增一个很薄的文件，例如：

```text
robot_ip_client.py
```

它的职责应严格限制在通信层，不要混入业务逻辑。

推荐只做下面几件事：

1. 建立和关闭 socket 连接。
2. 维护基础连接状态。
3. 把 6 维关节角格式化成字符串。
4. 把 `a`、`v` 拼接进命令。
5. 执行发送。
6. 在必要时打印少量调试日志。

不建议它负责：

1. 逆运动学。
2. 视觉坐标处理。
3. 行为树节点调度。
4. 夹爪动作逻辑。

### 7.3 推荐这个发送模块长什么样

后续实现时，建议接口尽量薄，类似下面这种风格：

```python
class RobotIPClient:
    def __init__(self, host, port, a=0.12, v=0.08, enabled=False):
        ...

    def connect(self):
        ...

    def close(self):
        ...

    def format_movej(self, joint_angles, a=None, v=None):
        ...

    def send_movej(self, joint_angles, a=None, v=None):
        ...
```

这样有几个好处：

1. 单一职责清楚。
2. 后面如果不是 URScript，也只需要改格式化函数。
3. 调用侧只需要关心“发这组角度”，不用关心 socket 细节。

### 7.4 在求解器里只加“可选发送能力”

推荐后面改 `GNNIKSolver.solve(...)` 时，不改核心求解过程，只增加可选参数。  
例如可以考虑这种思路：

```python
def solve(
    self,
    current_angles,
    target_pos,
    target_rot,
    ...,
    send_hook=None,
    send_policy=None,
):
    ...
```

其中：

1. `send_hook` 负责真正发送。
2. `send_policy` 负责决定“这一轮该不该发”。

这样设计的优点是：

1. 求解器仍然只负责求解。
2. 发送策略和通信实现可以独立替换。
3. 默认 `None` 时，现有代码完全不受影响。

### 7.5 为什么更推荐“回调/钩子”，而不是把 socket 直接写进 solver

如果直接在 `GNN/ik_solver_api.py` 里硬编码 socket：

1. 求解器会和真机通信强耦合。
2. 后面测试会变麻烦。
3. 仿真模式和真机模式容易互相污染。

如果只加一个回调：

1. `solve(...)` 不知道底层是不是 socket。
2. 你可以传“真发送回调”，也可以传“打印回调”。
3. 更容易做干跑测试。

这更符合“小改但不散结构”的目标。

### 7.6 在 `task_linear.py` 先做第一层接入，性价比最高

后续真正落代码时，我会优先从 [subtask/task_linear.py](c:\Users\admin\Desktop\GNN_solver\subtask\task_linear.py) 下手，而不是先改所有任务。

原因很简单：

1. 很多子任务最后都会走到线性插补。
2. 这里天然就是“外层低频发送点”。
3. 只要这里接上真机发送，很多动作链路就已经能联调。

推荐这里后续只加很少的参数，例如：

1. `robot_client=None`
2. `send_mode="none" | "outer_step"`

默认继续保持：

1. 不发送真机。
2. 只刷新仿真。

### 7.7 等外层稳定后，再决定是否改 `ik_solver_api.py`

如果外层发送稳定，再去改 [GNN/ik_solver_api.py](c:\Users\admin\Desktop\GNN_solver\GNN\ik_solver_api.py)。  
这里的改动也应控制在“增加可选发送出口”，而不是改求解算法。

推荐只做这类小改：

1. 在迭代末尾插入一个 `if send_hook is not None: ...`
2. 在发送前先做限位和策略判断
3. 记录上一次发送角度和时间

不推荐做这类大改：

1. 改模型输入输出结构
2. 改收敛条件
3. 改求解主循环层次

### 7.8 `a` 和 `v` 不要写死在多个文件里

真机阶段 `a`、`v` 会频繁调整，因此建议集中管理。  
可以先放在一个很薄的配置区，后续再决定是否抽配置文件。

初期建议范围：

1. 低速联调：`a=0.10 ~ 0.20`
2. 低速联调：`v=0.05 ~ 0.10`

推荐做法：

1. `robot_ip_client.py` 里有默认值。
2. demo 入口脚本允许覆盖。
3. 子任务内部不各自维护一份。

这样后面调参时只改一两个地方，不会全项目找常量。

### 7.9 推荐把“是否真机发送”放在入口层开关，而不是散落在任务内部

像 [subtask/task1_bt_demo.py](c:\Users\admin\Desktop\GNN_solver\subtask\task1_bt_demo.py) 这样的入口脚本，更适合放模式开关，例如：

1. `REAL_ROBOT_ENABLED = False`
2. `SEND_MODE = "none" | "outer_step" | "solver_iter"`

这样做的好处：

1. 一眼就知道当前在什么模式。
2. 便于回退。
3. 不会让每个子任务都带一堆通信分支。

### 7.10 建议预留但不要立刻做复杂实体

为了遵循“如无必要勿增实体”，后续不建议一开始就加：

1. 单独线程队列。
2. 复杂命令缓存器。
3. 多层中间抽象类。
4. 专门的任务调度服务。

如果后面联调证明简单直连不够，再逐步增加下面这些能力即可：

1. 重连机制。
2. 发送节流器。
3. 简单确认机制。
4. 日志记录。

也就是说，先把系统做“薄”，再看是否有证据需要变“厚”。

后续如果你让我正式开始改代码，我建议优先把改动控制在下面几个文件：

1. 新增 `robot_ip_client.py`
2. [GNN/ik_solver_api.py](c:\Users\admin\Desktop\GNN_solver\GNN\ik_solver_api.py)
3. [subtask/task_linear.py](c:\Users\admin\Desktop\GNN_solver\subtask\task_linear.py)
4. 某一个入口脚本，例如 [subtask/task1_bt_demo.py](c:\Users\admin\Desktop\GNN_solver\subtask\task1_bt_demo.py)

这样改动范围小、验证路径短，也最方便出问题时快速回退。

### 7.12 后续实现时不建议碰的文件

在真机本体联调阶段，下面这些尽量不动：

1. `run_vision.py`
2. GNN 模型结构文件
3. 数据集和训练脚本

因为这些内容和“先把 IP 控制真机打通”不是同一层问题。  
先把执行链路打通，再考虑更上层的任务闭环，会更稳。

## 8. 推荐的后续改造顺序

建议严格按下面顺序做。

### 第一步：只打通最小闭环

目标：

1. 上位机能通过 IP 连到真机。
2. 能手动发送一条固定关节角字符串。
3. 真机能低速执行。

此阶段不要接行为树，不要接视觉，不要接夹爪。

### 第二步：把发送逻辑接到现有项目，但仍保留仿真

目标：

1. 保留 MuJoCo 画面。
2. 同时把求得的关节角通过 IP 发给真机。
3. 验证仿真动作和真机动作是否基本一致。

此阶段建议先只接 `task_linear.py` 外层步进发送。

### 第三步：再尝试下沉到 solver 内部迭代发送

目标：

1. 在 `GNNIKSolver.solve(...)` 每轮迭代结束发送一次角度。
2. 加节流、防堆积、异常重连。

如果这一阶段真机表现不稳定，应立即回退到第二步的“外层插补发送”。

### 第四步：最后才考虑夹爪与抓取任务

目前先不做。  
等夹爪型号、开闭量、力度、安装尺寸明确后，再单独加一个夹爪控制层。

## 9. 推荐的字符串格式

如果是真机 UR10e，推荐直接发送：

```text
movej([q1,q2,q3,q4,q5,q6], a=0.15, v=0.08)\n
```

具体建议：

1. 角度单位保持弧度，与你当前项目一致。
2. 小数保留 6 位即可。
3. 每条命令都带换行，便于作为完整字符串发送。
4. `a`、`v` 在初期真机实验保持较低。

示例：

```text
movej([2.000000,-1.570000,1.570000,0.000000,1.570000,0.000000], a=0.12, v=0.08)\n
```

## 10. 安全建议

真机阶段一定要把安全项写进实际执行流程。

### 10.1 必须做的限制

2. `a`、`v` 初期保持低值。

4. 增加工作空间安全区域检查。

1. `GNN/ik_solver_api.py` 里已有 `CUSTOM_JOINT_RANGES`。
