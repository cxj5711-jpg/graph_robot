1. 这次修订的核心纠偏

上一版方案里，“不要把 solver_iteration 当真机轨迹”这个结论本身没有错，但表达得还不够贴近你的真实目标。你强调得非常对：GNN 的迭代求解是核心，不是要把迭代去掉，而是要在保持迭代的前提下，甄别哪些迭代结果应进入真实机械臂。

因此，这次方案不再把问题表述成“要不要迭代”，而是重新表述成下面这句话：

保留 GNN 迭代求解全过程，但把“求解过程中的内部状态”和“应该发送给真机的稳定命令点”区分开。

这意味着：

  1. GNN 仍然每步迭代，仍然进行误差修正、步长裁剪、关节限位和 MuJoCo 前向验证。
  2. 只是并非每一个内部迭代状态都要出现在 TCP 命令流中。
  3. 不同子任务对“该发哪一层结果”的要求不一样，不能一刀切。

2. 当前代码里已经存在的三层结果

你当前仓库实际上已经天然存在三层“角度结果”，只是之前还没有把它们严格命名并用于真机发送策略。

层级 当前代码中的来源 语义 是否适合直接发给真机
内层迭代解 GNNIKSolver.solve(...) 循环中的每轮 angles 神经网络逆解器的内部逼近过程 默认不适合。它主要服务于收敛，不天然等价于物理轨迹。
单次求解最终解 solver_final 对应的最终 angles 对某个目标位姿的单次收敛结果 适合简单点到点子任务。
外层路径步最终命令解 task_linear、task_arc、task_screw 中每个外层步的最终命令角 路径型子任务对每个离散轨迹步给出的稳定输出 最适合直线、圆弧、旋拧这类需要轨迹连续性的任务。
真机发送策略的本质，并不是替换求解器，而是决定：当前这次动作，应该消费哪一层结果。

3. 现有代码中最关键的事实

tcp.py 已经证明 TCP 链路是可用原型

你上传的 tcp.py 已经给出了最小可用的发送方式：

ROBOT_IP = "192.168.125.12"
PORT = 30002

cmd = f"movej([{joint_str}], a=1.2, v=1.05)\n"
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect((ROBOT_IP, PORT))
sock.send(cmd.encode())
sock.close()

这说明第一版真机接入完全可以继续采用 URScript + TCP + 30002 的方式，只需要把这段逻辑内聚到 RobotIPClient。

Task3 已经具备最有价值的外层步进结构

subtask/task_screw.py 中，旋拧原语并不是“直接把 solver 内部迭代结果都拿来发送”，而是：

  1. 先生成每个螺旋步的目标位姿。
  2. 对该步目标位姿调用一次 GNN 迭代求解。
  3. 得到该步稳定解后，再叠加腕关节累计旋转。
  4. 最后把真正的 cmd_angles 交给 apply_motion_step(..., stage="linear_step")。

这正是 Task3 最应发送的层级。

4. 本次修订后的总原则

原则 A： 不改 GNN 迭代求解本体。
原则 B： 点到点子任务优先发送“单次求解最终解”。
原则 C： 路径型子任务优先发送“每个外层轨迹步的最终命令解”。
原则 D： solver_iteration 保留为调试通道，而不是默认真机通道。
原则 E： 如无必要勿增实体，只在现有 RobotIPClient、solve() 和各子任务调用点上做最小改动。

5. 详细发送策略：按子任务分类，而不是按项目整体一刀切

下面这张表是这次方案的核心。它回答的是：每个子任务最终应该从哪一层结果中取命令发给真机。

子任务 当前代码特征 建议发送来源 建议原因
Approach 一次 solve()，目标是预接近点 solver_final 它本质是点到点预位姿调整，通常不必把内部迭代过程下发。
Orient 一次 solve()，目标是姿态对齐 solver_final 本质也是单次收敛后的最终角度更有意义。
Reach 内部调用 task_linear.execute linear_step 接近抓取点时更关心笛卡尔路径，不建议简单压缩成一个终点关节动作。
Retreat 内部调用 task_linear.execute linear_step 从接触区撤离时保持路径可控更安全，第一版应保守处理。
Linear 20 个外层插补步，每步一次 solve() linear_step 真正该发的是“每个外层步收敛后的结果”，不是每轮内部迭代角。
Arc 30 个外层圆弧步，每步一次 solve() linear_step 圆弧本身就是路径型任务，外层步进结果才有物理意义。
Screw 90 个螺旋步，每步一次 solve()，再叠加腕关节累计旋转 linear_step Task3 的关键不是发送内层解，而是发送每个螺旋步最终形成的 cmd_angles。
Hover 保持当前位姿 不发新的关节运动命令 它不产生新轨迹，只是停留。
Grasp / Release 当前是 MuJoCo 执行器控制 后续由独立夹爪层处理 不应和机械臂关节命令流混成一层。
这里最值得注意的是：Reach 和 Retreat 虽然看起来“简单”，但由于它们已经被实现成线性路径原语，第一版真机方案应继续按路径型任务对待，而不是贸然压缩成单条终点关节动作。

6. Task3 为什么不能简单“只取首尾”

你的判断是对的。Task3 的旋拧阶段不能像部分简单子任务那样只取首尾关节角，因为它不是普通的点到点动作，而是一个“沿轴向进给 + 绕同轴旋转”的耦合过程。

6.1 当前 task_screw.py 已经体现了正确的几何结构

  1. 由 SCREW_STEPS 把整段旋拧离散成多个螺旋步。
  2. 每一步先生成 interp_pos 和 interp_rot。
  3. 再让 GNN 对这一小步目标位姿进行完整迭代求解。
  4. 得到 solved_angles 后，还会额外累计腕关节角度，形成真正的 cmd_angles。
  5. 最后发送的是 cmd_angles，不是任意一个中间内部解。

6.2 Task3 最应该发送的是什么

对 Task3 来说，最该下发给真机的是：每个螺旋外层步收敛后形成的最终命令角 cmd_angles。

6.3 Task3 最不该发送的是什么

最不该做的是把 solve() 内部每一轮迭代角都直接发出去。因为那样发出去的不是“螺旋轨迹步”，而是“神经网络内部逼近轨迹”，两者语义不同。

6.4 对 Task3 的结论

  1. 不砍 GNN 迭代。
  2. 不把内部迭代过程当作默认真机轨迹。
  3. 保留每个螺旋步一次完整求解。
  4. 发送每个螺旋步的最终命令角。

7. 对简单子任务和路径子任务的统一解释

你提出的“某些简单子任务可以直接用 movej 取任务首尾关节角度”是成立的，但要加一条边界：

只有当子任务本质上是自由空间点到点动作时，才建议用单次求解最终角度；只要任务本质上是在近目标区域保持几何路径，就应继续保留外层路径步发送。

可优先当作点到点的

  • Approach
  • Orient
  • 后续若脱离障碍区的某些“回到安全位姿”动作

应保留路径步发送的

  • Reach
  • Retreat
  • Linear
  • Arc
  • Screw

8. 最小代码改动方案

遵循“如无必要勿增实体”，本次建议不增加新服务、不新增独立调度器、不重写求解器架构，只在现有类和调用点上做最少的结构化改动。

8.1 第一处改动：RobotIPClient 复用 tcp.py 变为真正的 TCP 发送器

这一步不是新架构，只是把你已经验证过的 tcp.py 通信逻辑内聚到现有桥接层中。

connect()
  -> socket.socket(AF_INET, SOCK_STREAM)
  -> socket.connect((host, port))

send_movej(joint_angles)
  -> command = format_movej(joint_angles) + "\n"
  -> socket.sendall(command.encode("utf-8"))
  -> optional: mirror the same command to output_path as a log

close()
  -> socket.close()

建议默认直接使用你已有的参数：

host = "192.168.125.12"
port = 30002
唯一与 tcp.py 不同的建议是：不要每发一条命令就重新 connect/close 一次。 对多步路径任务来说，应该在任务开始前建立连接，在整个行为树运行期间复用同一个 socket，结束时统一关闭。

8.2 第二处改动：给 solve() 增加“是否发内部命令”的最小控制

为了支持“简单任务发 solver_final，路径任务发 linear_step，但两者不重复发送”，需要在 solve() 层面增加最小的发射控制。

推荐增加两个可选控制，不必引入复杂新类型：

emit_solver_iteration: bool = False
emit_solver_final: bool = True

这样就可以做到：

使用场景 emit_solver_iteration emit_solver_final 外层是否再发 linear_step
点到点子任务 False True 否，避免重复
路径型子任务 False False 是，由外层步最终结果统一发送
调试内部迭代 True 可选 通常否

这一步非常关键，因为它能避免“solver_final 发一次、外层 linear_step 又发一次”的重复命令问题。

8.3 第三处改动：保持现有 stage 概念，但让其真正服务于任务甄别

不需要新建复杂的消息实体，只需要让当前已有的 stage 概念更加清晰：

  • solver_iteration：内部求解调试通道。
  • solver_final：简单点到点子任务的最终解通道。
  • linear_step：路径型子任务每个外层步的最终命令通道。

8.4 第四处改动：由各子任务选择发送层级，而不是全局只设一个模式

目前 RobotIPClient 只接受单个 send_mode 字符串，这对混合任务不够用。要保持简洁，可以把它扩展为“允许哪些 stage 被发送”的集合，但接口仍可兼容现有字符串模式。

enabled_stages = {"solver_final", "linear_step"}

这样并不是引入复杂新实体，只是把“一个字符串”扩成“一个很小的集合”。

10. 推荐的真实发送配置

在第一版真实代码中，推荐采用如下默认策略：

transport = "tcp"
host = "192.168.125.12"
port = 30002
enabled_stages = {"solver_final", "linear_step"}

# 点到点子任务
solve(..., emit_solver_iteration=False, emit_solver_final=True)
apply_motion_step(..., stage=None)

# 路径型子任务
solve(..., emit_solver_iteration=False, emit_solver_final=False)
apply_motion_step(..., stage="linear_step")
这个配置已经能满足你现在的核心诉求：
  1. GNN 仍然迭代求解。
  2. 简单任务可以只发最终收敛解。
  3. Task3 可以继续按每个螺旋小步发外层最终命令角。
  4. 整个结构没有被重写，只是在现有架构里把“发哪一层结果”理顺了。

11. 第一阶段不建议做的事

  1. 不引入 RTDE 和 servoj 作为第一版主方案。
  2. 不改 GNN 训练和迭代求解主逻辑。
  3. 不把 socket 代码散落到每个子任务里。
  4. 不新增独立通信进程、消息队列、中间件。
  5. 不把夹爪协议提前和机械臂关节流混在一起。
第一版的目标很明确：先把“甄别性发送 + 真机 TCP 下发”打通。只要这一版结构清晰，后面无论接夹爪、加状态反馈，还是进一步做更细的轨迹控制，都会顺很多。

12. 最终结论

这次修订后的最优方案不是“减少 GNN 迭代”，而是“保留 GNN 迭代，但把真机发送点前移到更有物理意义的层级”。具体来说：

  1. 简单点到点子任务，发送单次 solve() 的最终收敛解。
  2. 直线、圆弧、旋拧等路径型子任务，发送每个外层轨迹步最终形成的稳定命令角。
  3. Task3 的旋拧阶段，发送的是每个螺旋步的最终 cmd_angles，而不是内部迭代角。
  4. 真实 TCP 通信直接复用你已有的 tcp.py 方式，接入到 RobotIPClient 即可。
  5. 整个实现保持最小增量，不重写求解器，不重写行为树，不引入额外复杂实体。

13. 下一步真实代码改造顺序

  1. 先改 robot_ip_client.py，把 TCP 发送打通，保留日志镜像。
  2. 再改 GNN/ik_solver_api.py，加入最小的发射控制参数。
  3. 然后改 task_approach.py、task_orient.py、task_linear.py、task_arc.py、task_screw.py 的发送层级选择。
  4. 最后在三个行为树入口脚本中接入真实 IP、端口和启用配置。
