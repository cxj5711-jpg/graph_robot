UR10e 真机实验后续规划与连续控制改进路线

本文档面向当前 GNN_solver 仓库的真实执行阶段，目标不是推翻现有 GNN 逆解和行为树结构，而是在保持当前架构边界的前提下，给出真机实验的后续推进方案、当前优先改进点，以及从离散 movej 逐步演进到更连续、更稳定且不易卡顿的控制路线。

文档判断依据来自当前仓库中的 robot_ip_client.py、GNN/ik_solver_api.py、subtask/task_linear.py、subtask/task_screw.py、三个行为树 demo 以及当前导出的 robot_command_output/*.txt。

生成日期：2026-04-28 | 当前仓库阶段：已具备 TCP 文本发送与阶段化命令发射能力，但尚未形成真正的连续闭环执行栈

1. 当前项目状态判断

当前代码已经具备“真机发送出口”，但还没有形成“真机连续控制闭环”。也就是说，你现在已经可以把关节目标以 URScript 风格字符串发出去，但还没有把真实机器人状态反馈、发送节拍控制、命令缓冲与连续跟踪做成一条稳定链路。
层级 当前仓库事实 对真机实验的含义
发送适配层 robot_ip_client.py 已支持 TCP 发送与本地日志镜像，合法阶段为 solver_iteration、linear_step、solver_final。 发送边界已经抽出来了，这是后续改造成真机执行层的最好入口，不需要重写高层任务脚本。
求解器层 GNN/ik_solver_api.py 中的 emit_command() 能按阶段发命令；solve() 已支持开关 emit_solver_iteration 和 emit_solver_final。 说明当前架构已经考虑了“求解过程是否发命令”的问题，后面要做的不是乱加发送点，而是制定更严格的阶段策略。
插补执行层 subtask/task_linear.py、task_arc.py、task_screw.py 在循环内调用 solver.solve(..., emit_solver_iteration=False, emit_solver_final=False)，再用 apply_motion_step(..., stage="linear_step") 发一步。 这是一个正确的现阶段设计，说明你已经在避免“每个 solver 内部迭代都往真机发”的极端做法。
Primitive 边界 task_approach.py、task_orient.py 使用 emit_solver_final=True，成功后仅把姿态/位置应用到仿真，不在 apply_motion_step() 中重复发阶段命令。 说明当前 primitive 完成点发送一次最终姿态的思路是清晰的，不存在明显的阶段边界重复下发。
任务入口 task1_bt_demo.py、task2_bt_demo.py、task3_bt_demo.py 默认都配置为 COMMAND_SEND_STAGES = ("solver_final", "linear_step")，目标端口为 192.168.125.12:30002。 说明现阶段是真机“离散点 + 插补步”混合发送，而不是完全连续的在线伺服控制。
当前日志密度 当前导出的命令数量约为：Task1 42 条、Task2 92 条、Task3 164 条。 仅按当前外层插补与阶段发送，命令量已经不低；如果贸然切到 solver_iteration 级发送，负载会迅速放大。
从仓库现状看，最危险的误区不是“没有真机接口”，而是“误以为只要把发送频率继续拉高，就等于更连续”。实际上更连续控制需要同步解决反馈、节拍、缓冲、跟踪误差和执行语义，而不仅是多发几条 movej。

2. 你现在最需要改进的点

2.1 先补齐真机实验基线，而不是先追求高频

  1. 完成 TCP、Payload、TCP 工具坐标、零位、末端工具长度、工作空间边界和安全 home 点的统一校准。
  2. 明确“谁拥有控制权”：示教器、外部上位机、还是后续 RTDE/驱动程序，避免实验时出现控制源切换和状态不一致。
  3. 把故障恢复流程标准化：网络断开、急停、保护停、超限、抓取失败后如何回到安全姿态，必须先写成实验 SOP。

2.2 发送链路必须从“能发”升级到“可观测”

  1. 区分 commanded_q 和 actual_q。当前很多子任务把“上一轮求解出的关节角”直接当成下一轮起点，这在仿真中成立，但真机一旦有跟踪滞后就会产生模型与真实机械臂脱节。
  2. 增加统一的带时间戳日志，至少记录：发送时刻、阶段、目标关节角、加速度、速度、发送结果、真机反馈状态。
  3. 后续要接入实时反馈变量，优先记录：actual_q、actual_TCP_pose、actual_qd、speed_scaling、robot_mode、safety_status。

2.3 运动层要从“求解器输出即执行目标”升级为“执行器二次整形”

  1. GNN 求解器负责给出可达方向和关节更新，不应该直接承担所有真机轨迹平滑责任。
  2. 执行层应增加“再采样 / 降频 / 阈值过滤 / 速度加速度限幅 / 过期命令丢弃”的二次整形。
  3. 螺旋装配阶段不能简单理解为“更多直线步”。它天然更敏感，更需要局部连续伺服、缓冲队列与实时反馈。

2.4 实验评价必须从“到没到”升级为“平不平、稳不稳、跟不跟得上”

  1. 除了最终位姿误差，还要记录发送周期抖动、任务耗时、实际跟踪误差、保护停次数和异常中断率。
  2. 连续控制的核心指标不是命令条数，而是机器人是否平滑、是否无停顿、是否没有明显追赶与回拉。
  3. 装配类任务还要额外记录末端速度波动、螺旋阶段失败步比例和旋进阶段的关节 6 跟随稳定性。

3. 为什么当前真机容易出现卡顿

原因 A：接口频率和意图节拍不匹配

当前 demo 默认发往 30002。根据 UR 官方“Primary / Secondary / Realtime Interface”教程，30002 的 Secondary Interface 是 10 Hz，更适合低优先级和非关键命令；而当前仓库里 task2 与 task_arc 常见步进延时为 0.02s，等价于理论 50 Hz 节拍，task1 / task3 的 0.05s 也接近 20 Hz。

这意味着上位机的“想发多快”和当前端口“适合承载什么节拍”并不一致，卡顿、堆积和非平滑执行是高概率事件。

原因 B：连续发送 movej 本质上仍是离散重规划

你现在发出的每一条命令都是完整的关节空间目标。即使它们来自很密的插补点，对控制器而言也更像一串离散目标，而不是一条连续伺服流。

这会造成机器人频繁重定向、重新限速或在段与段之间出现轻微停顿，尤其是在目标点间距不均匀、姿态变化又比较大的时候更明显。

原因 C：当前求解起点主要来自“命令状态”而非“真实状态”

task_linear.py、task_arc.py、task_screw.py 都在循环中把上一轮求解结果继续传回下一轮 solver。对仿真来说没问题，因为仿真状态会立刻同步。

但真机存在执行滞后时，下一轮求解如果继续从“想象中的已到达姿态”出发，就会导致上位机比机器人更超前，最终表现为追赶、补偿、回拉和抖动。

原因 D：当前没有显式缓冲与拥塞控制

现在的发送链路没有“队列长度上限、过期命令覆盖、最小发送周期、角度变化阈值、反馈滞后保护”等机制。

一旦网络抖动、控制器忙、或装配阶段局部难解，命令就可能积压；积压之后再执行，就会让机器人动作看起来像“卡一下，再补一下”。

结合当前参数，风险最大的场景是把 solver_iteration 直接开放到真机。以当前仓库参数估算：Task2 的直线段理论上可放大到 20 x 50 = 1000 次更新机会；Task3 螺旋段若在每个螺旋步内部都开放 solver 迭代发射，则理论上可到 90 x 60 = 5400 次更新机会。这个量级已经不是“更连续”，而是“极易把执行链路冲垮”。

4. 更连续控制的推荐路线

推荐原则：先把离散命令做稳，再做缓冲连续，最后再做真连续伺服。 不建议从当前 movej + 30002 直接一步跳到“每次 solver 迭代都实时下发”。
阶段 目标 推荐控制方式 这一阶段要解决的问题
P0：真机基线期 先确保“能稳定到位、能安全恢复、参数方向完全正确” 继续使用当前 movej，但只保留低速、低频、外层关键点发送 校准、限位、方向验证、断线恢复、急停恢复、日志打通
P1：受控连续期 在不改核心架构的前提下显著减轻停顿 linear_step 为主，加入发送限频、角度阈值、队列覆盖、真实状态回读 解决堆积、减少重复命令、避免“上一条未执行完下一条又来了”
P2：缓冲执行期 让执行器具备轨迹感，而不只是“收一条动一条” PC 侧生成短时间窗轨迹片段，机器人侧或执行线程侧做缓冲消费 解决段间停顿、延迟抖动、螺旋阶段的不平顺
P3：真连续伺服期 实现更接近连续关节跟踪的控制效果 后续切换到 RTDE / 反向接口 / servoj 风格闭环方案 解决高密度 setpoint 跟踪、实际状态闭环、平滑性与时效性统一问题

4.1 P0 阶段：先把当前 movej 流程做稳

  1. 保留当前 ("solver_final", "linear_step") 设计，不打开 solver_iteration 真机发送。
  2. 把所有任务的实验速度先统一降到一个保守档位，先做 Task1 和 Task2，不直接上 Task3 装配。
  3. 所有直线、圆弧、靠近、对准动作先用“低速 + 空载 + 大安全间隙”反复验证。
  4. 每次实验记录最少一份发送日志和一份真机状态日志，形成可回放问题样本。

4.2 P1 阶段：在现有架构中做第一轮防卡顿升级

  1. 给 RobotIPClient 增加最小发送周期 min_send_interval，优先从 5 到 10 Hz 量级做实验，而不是追求 20 到 50 Hz。
  2. 加入角度变化阈值。若本次目标相较上次发送的最大关节变化不足阈值，则直接跳过。
  3. 发送线程与计算线程解耦。求解器只生成目标，发送器决定“什么时候发、发哪条、是否覆盖旧命令”。
  4. 给队列设很小的上限，例如 1 到 3 个目标。新目标比旧目标更新时，允许覆盖未执行旧目标，避免堆积。
  5. 每个外层动作步开始前，用 actual_q 或最新可信反馈重置下一轮求解起点，至少定期重同步，而不是无限沿用命令态。

4.3 P2 阶段：让执行器具备短时缓冲能力

  1. 把“一个 primitive = 一串时间有序的 setpoint 片段”而不是“一步求解出一个点就立即发送”。
  2. 对片段做二次整形：速度限幅、加速度限幅、局部平滑、关节增量裁剪、段末过渡抑制。
  3. 螺旋装配单独处理，不再简单复用普通直线步调度。螺旋阶段更适合固定周期、固定步长、可回读状态的局部控制器。
  4. 当真机反馈显示跟踪误差增大或 speed_scaling 下降时，自动降低发送率或暂停新目标注入。

4.4 P3 阶段：真正想做到“连续”时，建议切换到更合适的控制接口

如果你的目标是明显减少段间停顿，并实现接近连续跟踪的控制效果，那么长期来看不应继续把高密度 setpoint 都压在 30002 + movej 上。更合理的方向是引入 RTDE 与伺服式控制链路。

  1. UR 官方 RTDE 文档说明其默认可做同步数据交换，并提供 actual_q、actual_TCP_pose、actual_qd、speed_scaling、safety_status 等输出变量，这正好满足你做闭环与观测的需要。
  2. UR 官方驱动参数文档说明 servoj 相关的 gain 和 lookahead_time 会影响“平滑性”和“延迟”的平衡，这比单纯堆更多 movej 更接近你想要的“连续但不抖”。
  3. UR 官方示例还强调，在进入连续写关节命令的循环前，应先读取 actual_q 作为目标缓冲初始化，这个思路很值得你后面迁移时照搬。

5. 防卡顿专项改进建议

发送节拍控制

  1. 增加最小发送周期。
  2. 加入角度死区阈值。
  3. 加入速度/加速度门控。
  4. 队列只保留最新未执行目标，避免越积越多。

反馈闭环

  1. 周期性读取 actual_q 和 actual_TCP_pose。
  2. 当 |actual_q - commanded_q| 超阈值时暂停继续推新点。
  3. 当 speed_scaling 持续下降时自动降频。
  4. 保护停或断连后立即清空发送队列。

轨迹整形

  1. 保持笛卡尔步长均匀，而不是只按固定步数切分。
  2. 接近目标时自动降速、降频、放宽求解微调发送。
  3. 螺旋段使用固定周期的局部控制器，不与普通线性步复用完全相同策略。
  4. 必要时对目标关节序列做轻量低通平滑。

异常与恢复

  1. 网络中断后不要自动续发旧队列。
  2. 任何保护停后，重新标定“当前真实关节角”再继续。
  3. 每个 primitive 结束都检查真机实际状态，而不是只相信上位机内部变量。
  4. 把“恢复到安全 home 点”的动作做成标准流程，而不是临时人工处理。
一个非常关键的工程原则是：连续控制的平滑性主要来自“反馈与节拍的一致”，不是来自“发送更密”。 如果没有反馈同步，再密的 setpoint 也可能只是在给控制器制造拥堵。

6. 推荐实验顺序

实验阶段 建议任务 主要目的 通过标准
E1：通信空跑 单独发送几组固定 movej 验证 IP、端口、远程控制模式、命令字符串格式、关节顺序 无保护停；到位方向正确；日志可复现
E2：低速单 primitive 先做 orient、approach 验证 solver_final 类命令的最终到位稳定性 无明显抖动；末端方向正确；误差稳定
E3：线性步执行 Task1 / Task2 中的 linear_step 验证外层插补密度与真机跟随能力 段间无明显停顿；没有命令堆积迹象
E4：弧线与扫描 Task2 圆弧扫描 验证姿态变化叠加位移时的平滑性 路径连续；视角稳定；轨迹没有回拉
E5：螺旋装配预试 Task3，先空载、再轻接触 验证螺旋阶段是否需要单独控制器 旋进阶段无剧烈抖动；失败步比例可控
E6：连续控制升级验证 P1 / P2 / P3 新链路 比较离散点、缓冲轨迹、伺服流三类方案差异 平滑性、耗时、误差、稳定性均优于旧方案

7. 建议记录的实验指标

指标 含义 为什么重要
发送周期均值 / 方差 上位机发命令的平均节拍与抖动 可以直接判断是否存在不稳定发送与拥塞
actual_q - commanded_q 真实关节角与目标关节角的偏差 是连续控制是否“跟得上”的核心指标
actual_TCP_pose 误差 真实末端位姿与任务目标的偏差 验证最终效果，不只看关节层
speed_scaling 轨迹限速器缩放系数 可以反映控制器是否在被迫降速或保护性限速
保护停 / 异常中断次数 安全层面失败统计 是能否进入正式实验的重要门槛
任务总耗时 从启动到完成的总时间 便于比较不同控制方案的执行效率
装配失败步比例 螺旋阶段局部步求解或跟踪失败占比 能直接反映装配连续性是否足够

8. 对当前代码层的后续改造建议

文件 建议职责 后续改造方向
robot_ip_client.py 保持为执行适配层 增加最小发送周期、最新命令覆盖、发送队列、失败重连、日志时间戳、状态反馈接口
GNN/ik_solver_api.py 保持为求解器 不要让它变成高频网络发送器；只保留受控回调与阶段发射开关
subtask/runtime_bridge.py 统一仿真状态应用与阶段命令发射 增加阶段元数据、命令编号、记录 hooks，后面可接入真机状态同步
subtask/task_linear.py 保持外层轨迹步组织 改为按实际距离自适应分段，并支持从真实反馈重置 solver 起点
subtask/task_screw.py 保持装配局部控制逻辑 后续优先单独升级，最有必要率先切换到缓冲或伺服风格控制
subtask/task1_bt_demo.py 等 保持任务级参数入口 集中管理实验预设：安全档、低速档、连续控制档、装配档
这条改造思路的优点是：你可以先保留当前论文里的整体结构不变，只替换“执行出口”和“反馈闭环”，不会把 GNN、行为树、子任务 primitive 全部打散重做。

9. 最推荐的近期推进顺序

  1. 先在当前架构上完成 P0：真机基线验证、日志打通、安全恢复流程。
  2. 再做 P1：给 RobotIPClient 增加限频、阈值、队列覆盖和反馈重同步，先把 Task1 / Task2 跑顺。
  3. 之后单独攻关 Task3：把螺旋装配阶段从普通 movej 分段思路里拆出来，建立更专门的局部执行器。
  4. 当你明确需要“更连续且更平滑”时，再进入 P2 / P3，逐步迁移到带反馈的缓冲执行，最终再考虑 RTDE + servoj 风格链路。
简短结论：当前最优策略不是立刻开放 solver_iteration 真机发送，而是先把现有 linear_step + solver_final 做稳，再逐步引入限频、反馈、缓冲，最后才进入真正的连续伺服控制。
